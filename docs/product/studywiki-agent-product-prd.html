🧠 StudyWiki-Agent

「把文档变成你的第二大脑」— 本地知识库 AI Agent

产品设计文档 · v0.4 · 2026-06-22 · 作者: AI Agent

一句话定位：类 Notion × 类 Anki，但 全部自动——上传课件 PDF，AI 自动写卡片、出题、跟踪掌握度。纯本地，不上云。

1. 背景与目标

解决什么现实问题？

学生在复习时面临三个核心痛点：

  • 整理成本高：听课笔记 + 课本扫描件 + 老师 PPT → 散落在不同文件里，复习时翻箱倒柜
  • 被动学习：看了一遍课件就以为自己懂了，但一考试就忘。缺少主动测试和间隔重复机制
  • 知识孤立：同一门课的知识点之间没有连接，跨课程的交叉理解更难建立

现有工具（Notion、Obsidian、Anki）都需要 手工 整理、写卡片、出题——用户没这个时间。

成功标准（非技术指标）

  • DAU 1000 原型上线后 3 个月内
  • 次日留存 ≥ 30%——说明「上传 → 复习 → 出题」的闭环成立
  • 人均卡片 ≥ 50 张——说明用户愿意持续导入资料
  • Quiz 完成率 ≥ 60%——AI 出题质量够好才会有人做

3. MVP 定义

Must-have（v0.4 必须交付）

  • P0 PDF/Word/MD 上传 + 自动提取知识卡片——核心价值，不通此路其他免谈
  • P0 知识卡片展示 + 超链接跳转——类维基百科的阅读体验
  • P0 Quiz 出题 + AI 评分——3-5 道简答题，LLM 打分+反馈
  • P0 掌握度进度条——每张卡片的 mastery_pct 可视化
  • P0 多知识库隔离——不同学科/项目的数据不混在一起
  • P0 前端界面可用——当前基于 PicoCSS + 手写样式，后续迁移至统一组件库。可交互，无崩溃
  • P0 测试集 ≥ 100 个用例——覆盖核心 CRUD + Quiz + Agent 流程

Nice-to-have（v0.5+）

  • P1 间隔重复（SM-2 算法）——到期自动提醒复习
  • P1 知识图谱可视化——卡片之间的关联关系用力导向图展示
  • P2 移动端适配/ PWA——手机上看卡片
  • P2 多端同步（WebDAV / iCloud / 自建服务器）
  • P2 浏览器插件——选中网页文本直接导入知识库
  • P3 团队协作——共享知识库 + 权限管理

明确不做什么

  • ❌ 不做笔记编辑器——用户 Notion/Obsidian 写笔记，StudyWiki 只做「知识提取 + 复习」
  • ❌ 不做云端同步（v0.x）——隐私第一，用户数据在本地
  • ❌ 不做手写识别——OCR 只做印刷体，手写笔记不在范围内
  • ❌ 不做实时协作——那是 Notion 和 Google Docs 的事
  • ❌ 不做 App Store 发布——Windows 桌面端 + 浏览器，足够了

5. 设计理念

体验关键词

⚡ 即刻 🧘 专注 🔗 连接 🔒 私有

核心交互原则（≤ 3 条）

  1. 上传即所得：用户上传 PDF 后，不需要任何手动操作——卡片自动生成、关联自动建立。如果 5 秒内没有看到结果，就是 bug。
  2. 浏览即复习：查看卡片时自动触发 recall 模式。看内容 → 立刻 Quiz → 记住。不走「先看后做」两步。
  3. 知识可入亦可出：每张卡片都可以导出为 Markdown / Anki 格式。知识不锁在系统里。

视觉风格方向

当前基于 PicoCSS（https://picocss.com）提供基础样式，手写 utility 类补充布局：

  • PicoCSS 为基础——主背景 #f8f9fc，组件圆角 8px，视觉干净不刺眼
  • 色彩节制——主色 #4299e1（温和蓝），辅色 #48bb78（完成绿），强调色 #ed8936（注意橙）
  • Typography 优先——内容区用大行高（1.7+）、适中的字号（15px），减少阅读压迫感
  • 暗色模式——默认跟随系统，可在 Settings 切换

6. 本轮优化方向（v0.4 → v0.5）

基于当前版本的问题和用户反馈，确认以下优化优先级：

优先级优化项当前状态目标预估工作量
P0 前端组件化 — 替换手写样式为组件库 原生 JS + PicoCSS + 手写 utility 类（48+ 行）。弹窗/按钮/卡片等组件不统一，滚动/布局存在多处 bug 引入组件库替换全部手写 UI 组件（待选型确认：EUI-NEO / DaisyUI / 其他）。砍掉手写 utility 类和 inline style。确保侧边栏可滚、Modal 可关、卡片超链接工作 中（2-3 天）
P0 测试集补全 ≥ 100 用例 现有 ~73 个测试用例，缺少：KB 切换隔离性测试、上传作业 enrich 测试、Agent ReAct 多轮测试、前端集成测试 提升到 100+ 个用例，覆盖所有 API 端点和核心 Agent 流程。增加 KB 隔离测试（确认上传到 KB A 的数据不会出现在 KB B） 中（2 天）
P1 PDF 解析鲁棒性 QQ 浏览器转格式的 PDF 导致整页卡死（已加超时，但后续处理未验证） 加入图片型 PDF 的整页 OCR fallback + 空页跳过 + 解析进度实时推前端 小（1 天）
P1 加载/空/错误状态全覆盖 部分场景（上传中、Quiz 加载中、KB 切换中）缺少 spinner；网络错误无 Retry 按钮 所有 async 操作显示 spinner + 超时提示 + 失败重试按钮 小（0.5 天）
P2 间隔重复（SM-2） 掌握度只有「百分比」，没有「下次复习时间」 基于 mastery_pct + 最后一次 Quiz 时间，计算下次复习日期。侧边栏显示「待复习 (3)」 中（1.5 天）

7. 风险

市场风险

  • 竞品——RemNote、Mem.ai、Quizlet 都在做类似的 AI 辅助学习工具。差异点在于 StudyWiki 是 纯本地 + 开源，适合对隐私敏感的用户和企业。
  • 会不会没人用？——最大的风险是「上传后发现卡质量差，然后就弃用了」。解决方案：在 MVP 阶段就强制保证 LLM Card 生成质量（prompt 工程 + 聚合提取 + 去重）。

运营风险

  • 版权合规——用户上传的课件可能包含受版权保护的内容。声明：数据在本地，不上传服务器，用户自行承担版权责任。
  • OCR 隐私——如果使用云 OCR API（不是 Tesseract 本地），用户的文档内容会被发送到第三方。解决方案：默认使用本地 Tesseract，OCR 数据不出本机。

增长风险

  • 冷启动——空知识库没有任何可看的内容，用户上传后需要等待数分钟才能看到第一张卡片。解决方案：首次启动时导入示例知识库（5-10 张卡片），让用户立刻感受到价值。
  • 传播渠道——学生群体的传播主要靠口碑和社交群（QQ/微信群）。需要设计分享功能（导出卡片为图片 → 分享到群）。

变现路径

  • 短期（v0.4 ~ v0.6）：免费开源，通过 GitHub Star 和社群积累用户。
  • 中期（v0.7+）：推出 Pro 版（多知识库无限、间隔重复、团队协作），按年订阅 ¥99/年。
  • 长期（v1.0+）：B 端按席 License + 私有部署服务费。
