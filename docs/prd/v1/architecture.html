2. 模块详解

M1 · 前端 UI 组件化 P0

当前状态：PicoCSS v2（CDN）+ 48+ 行手写 utility 类（.flex、.gap-*、.mt-* 等）+ 原生 JS 组件。vendor/ 目录存在 alpinejs / daisyui / tailwind / marked 文件但未加载。

目标：替换为统一组件库，消除所有手写工具类和 inline style。

组件替换清单：

当前实现目标组件位置
手写 navbar + PicoCSS组件库 header/navheader
手写 sidebar + 手写 +/- 交互组件库 sidebar + nav-item侧边栏
手写 card div + utility组件库 card卡片详情
手写 modal mask + display 控制组件库 modalQuiz/Edit/Settings
手写 button组件库 button (primary/ghost/error)全部按钮
原生 input/textarea/select组件库 input/select/textarea搜索/编辑/设置
手写 spinner组件库 spinner加载动画

迁移策略：一次一个组件替换，不 rewrite 整个文件。每替换一个组件后验证交互不坏。

M2 · 测试集补全 ≥ 100 P0

新增测试方向：

  • KB 隔离 + 并发上传测试
  • 作业 enrich 测试
  • Agent ReAct 多轮 + 工具调用测试
  • 前端集成测试（DOOM 存在性 + 交互）

当前覆盖（v1）：~73 个用例（8 个测试文件）。目标 ≥ 100。

M3 · PDF 解析容错 P1

改动点：

  • tools.py::_parse_pdf — 损坏 PDF 打开异常捕获，降级到 PIL+OCR
  • 单页 30s 超时（已有，验证有效）
  • 解析进度通过 _tasks 更新 + WebSocket 推送
  • DocumentScanner — 跳过空白页/封面/目录

M8 · WebSocket Chat & 双模式 Core

文件：bobanana/routes/chat.py（177 行）+ bobanana/agent_react.py（245 行）

核心组件：

  • ConnectionManager — WebSocket 连接管理，支持 connect/disconnect/send/broadcast
  • make_progress_callback — 进度事件回调，call_soon_threadsafe 从后台线程推送到 WebSocket
  • Ask 模式 — 知识库检索 + LLM 回答，不调用工具
  • Agent 模式 — CoT + ReAct 循环：Thought→Action→Observation→Final Answer，最多 6 轮

端点：ws://host:port/ws/chat

消息格式：

  • 接收：{type: "message", content: "...", data?: {mode: "ask"|"agent"}}
  • 发送：{type: "response"|"progress"|"error", content: "...", data?: {...}}

M9 · Agent 导入 Pipeline Core

文件：bobanana/agent.py（367 行）+ bobanana/tools.py（438 行）

三阶段流水线：

第1阶段: DocumentScanner 预扫描
↓  跳过空白页、封面、目录 → 输出有效区间 [(1,3), (5,8), …]
第2阶段: 智能提取 (3 线程并发)
├─ 短区间(≤2页) → 逐页 LLM 提取 (SYSTEM_EXTRACT)
└─ 长区间(≥3页) → 聚合后 LLM 提取 (AGGREGATE_EXTRACT)
第3阶段: 增量入库
   逐条 CardCreate → CardService.create_card (含去重 + 关联)

Homework 模式：上传作业后，提取的知识点匹配已有卡片（按标题/别名），将新内容 enrich 进原卡片。

LLM JSON 解析容错：_parse_llm_json 有 4 层 fallback：代码块剥离 → 单引号修复 → 尾逗号清理 → 截断补全。

M10 · Card CRUD Service Layer Core

文件：bobanana/service/card_service.py（312 行）+ bobanana/database.py（196 行）

架构设计：

CardService 单例
├─ 读操作（无锁）: get / list / search / categories
├─ 写操作（threading.Lock 序列化）
│  ├─ embedding 锁外预计算 (避免阻塞)
│  ├─ create / update / delete / batch_import
│  └─ deduplicate (cosine > 0.93 合并 + 删除)
├─ 同步方法 (供 Agent 线程): *_sync()
│  └─ asyncio.run() 跨事件循环
└─ 关联检测: _detect_and_link (锁外)
   └─ 标题/别名匹配 → 双向 related_cards 更新

DatabaseManager 单例
├─ ChromaDB 持久化 (HNSW 索引)
├─ CRUD / search / categories
├─ KB 隔离: kb_{id} 集合
└─ collection_lock: threading.Lock

嵌入模型：支持 openai / sentence-transformers 两种，通过 config.py 的 EMBEDDING_PROVIDER 切换。

6. 架构决策记录

ADR-1：选择 2+1 方案 (a) 按模块拆

理由：M1-M11 边界清晰，每个 task 对应一个模块，验收条件独立。

ADR-2：组件库通过 CDN 引入，不打包构建

理由：当前项目无前端构建工具链，CDN 方式最低侵入。

ADR-3：测试使用已存在的 pytest 框架

理由：已有 pytest 用例和 fixture，复用 conftest。

ADR-4：CardService 单例 + threading.Lock 序列化写操作

理由：ChromaDB 客户端非线程安全。锁外计算 embedding 可避免 CPU 阻塞其他写请求。跨事件循环通过 asyncio.run() 桥接（已知嵌套调用风险）。

ADR-5：Agent ReAct 使用本地循环而非外部 Agent 框架

理由：避免引入 LangChain/AutoGen 等重型依赖。6 轮循环硬限制 + 超时兜底，防止无限循环。

ADR-6：掌握度 / 历史使用进程内存储（P1 骨架）

理由：简化 MVP 实现。后续计划迁移到 ChromaDB metadata 或独立的 SQLite 表。
