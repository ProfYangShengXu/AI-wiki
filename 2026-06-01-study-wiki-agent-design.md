## 1. 项目概述

### 1.1 项目名称

**StudyWiki-Agent** — 基于 LangChain + LangGraph 的可增长可编辑本地网页 Wiki 知识库 AI Agent。

### 1.2 目标用户

需要复习期末 PPT 的大学生群体。

### 1.3 项目背景

大学生在期末复习时，面对大量 PDF/Word/MD 格式的课件资料，缺乏一个能够自动提取知识点、组织知识卡片、支持智能问答和知识库动态增长的工具。本项目旨在通过 AI Agent 技术，将散乱的课件文档转化为结构化的、可交互的本地 Wiki 知识库。

### 1.4 核心功能

| 功能 | 描述 |
|------|------|
| **文档解析与知识提取** | 接收 PDF/Word/MD 文档，逐页分析，自动提取知识点（名词、解释、案例、问题），生成关联知识卡片存入数据库 |
| **智能问答** | 用户通过前端与 Agent 实时对话，Agent 检索知识库并给出回答 |
| **知识卡片浏览** | DaisyUI 前端展示知识卡片，支持超链接跳转、目录导航、前进/后退、搜索 |
| **知识库增长** | 用户导入文件自动解析入库；用户手动要求 Agent 补全知识点，确认后保存 |
| **卡片修改** | 用户通过对话要求 Agent 修改已有卡片内容 |
| **网络搜索补充** | Agent 在解析文档发现信息不足时，自动调用网络搜索补充 |

### 1.5 技术栈

| 层次 | 技术 | 说明 |
|------|------|------|
| **后端框架** | Python 3.10+, FastAPI | Web 服务框架 |
| **AI 框架** | LangChain, LangGraph | Agent 工作流编排 |
| **AI 模型** | OpenAI GPT-4 / Ollama 本地模型 | LLM 推理 |
| **向量数据库** | ChromaDB | 本地轻量级向量存储 |
| **文档解析** | PyMuPDF (fitz), python-docx, markdown | PDF/Word/MD 解析 |
| **网络搜索** | DuckDuckGo Search (duckduckgo_search) | 知识补全 |
| **前端** | HTML + Tailwind CSS + DaisyUI + Alpine.js | 本地网页 UI |
| **嵌入模型** | sentence-transformers / OpenAI embeddings | 文本向量化 |

### 2.2 分层说明

#### 2.2.1 用户层 (Browser)

- 使用 DaisyUI 构建的本地网页前端
- 通过 HTTP REST API 与后端通信获取卡片数据
- 通过 WebSocket 与 Agent 进行实时对话
- 提供卡片展示、目录导航、搜索、前进/后退、对话窗口等 UI 组件

#### 2.2.2 后端服务层 (FastAPI)

- **REST API**：提供知识卡片的 CRUD 操作接口
- **WebSocket**：管理 Agent 与用户的实时对话
- **文件上传接口**：接收用户上传的 PDF/Word/MD 文件

#### 2.2.3 AI Agent 层 (LangGraph)

- 使用 LangGraph 的 StateGraph 构建有状态工作流
- 包含文档解析、知识提取、卡片生成、问答检索、卡片修改、网络搜索补全等节点
- 工具集封装了数据库操作、文档解析、网络搜索等能力
- 记忆模块维护对话历史上下文

#### 2.2.4 数据层

- **ChromaDB**：存储知识卡片的向量表示和元数据
- **本地文件系统**：存储上传的原始文件和 ChromaDB 持久化数据

## 4. 数据库设计

### 4.1 ChromaDB Collection 设计

#### Collection: `knowledge_cards`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `str` (UUID) | 卡片唯一标识符 |
| `title` | `str` | 知识点名词（主标题） |
| `aliases` | `list[str]` | 别名列表（用于检索匹配） |
| `content` | `str` | 详细解释内容 |
| `examples` | `list[str]` | 案例列表 |
| `questions` | `list[str]` | 相关问题列表 |
| `category` | `str` | 知识领域分类 |
| `source_file` | `str` | 来源文件名 |
| `source_page` | `int` | 来源页码 |
| `related_cards` | `list[str]` | 关联卡片 ID 列表 |
| `created_at` | `str` (ISO 8601) | 创建时间 |
| `updated_at` | `str` (ISO 8601) | 更新时间 |

#### 向量嵌入

- **嵌入模型：** `text-embedding-ada-002` (OpenAI) 或 `all-MiniLM-L6-v2` (sentence-transformers)
- **嵌入字段：** `title` + `aliases` + `content` 拼接后生成向量
- **检索方式：** 余弦相似度搜索，返回 Top-5 或 Top-10

### 4.2 数据关系

```
知识卡片 (KnowledgeCard)
     │
     ├── 标题 (title) ────────────── 主名称
     ├── 别名 (aliases) ──────────── 同义词/简称，用于检索
     ├── 内容 (content) ──────────── 详细解释
     ├── 案例 (examples) ────────── 多个案例
     ├── 问题 (questions) ───────── 复习问题
     ├── 分类 (category) ────────── 知识领域
     ├── 来源 (source_file/page) ── 出处
     └── 关联 (related_cards) ───── 指向其他卡片的 ID 列表
```

### 4.3 卡片关联检测

在卡片入库时，Agent 自动检测 `content`、`examples`、`questions` 中是否出现其他卡片的 `title` 或 `aliases`，若匹配则建立双向关联：

- 在 `related_cards` 中添加关联卡片 ID
- 在前端渲染时，匹配到的名词自动转为超链接

## 5. API 设计

### 5.1 REST API 接口

#### 5.1.1 知识卡片 CRUD

| 方法 | 路径 | 说明 | 请求体 / 参数 |
|------|------|------|--------------|
| `GET` | `/api/cards` | 获取所有卡片列表 | `?category=xxx&page=1&limit=20` |
| `GET` | `/api/cards/{card_id}` | 获取单张卡片详情 | — |
| `POST` | `/api/cards` | 创建新卡片 | `{title, aliases, content, examples, questions, category}` |
| `PUT` | `/api/cards/{card_id}` | 更新卡片 | `{title?, aliases?, content?, ...}` |
| `DELETE` | `/api/cards/{card_id}` | 删除卡片 | — |
| `GET` | `/api/cards/search` | 搜索卡片 | `?q=关键词` |

#### 5.1.2 文件上传

| 方法 | 路径 | 说明 | 请求体 |
|------|------|------|--------|
| `POST` | `/api/upload` | 上传文件并触发解析 | `multipart/form-data: file` |

#### 5.1.3 目录与分类

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/categories` | 获取所有分类列表 |
| `GET` | `/api/cards/by-category/{category}` | 按分类获取卡片 |

#### 5.1.4 历史记录

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/history` | 获取浏览历史 |
| `POST` | `/api/history` | 记录浏览卡片 |

### 5.2 WebSocket 接口

| 路径 | 说明 | 消息格式 |
|------|------|---------|
| `/ws/chat` | Agent 实时对话 | JSON: `{type: "message"|"command", content: "..."}` |

**WebSocket 消息类型：**

| 类型 | 方向 | 说明 |
|------|------|------|
| `message` | 用户 → Agent | 用户提问或指令 |
| `response` | Agent → 用户 | 回答文本 |
| `card_preview` | Agent → 用户 | 新卡片预览（待确认） |
| `card_update` | Agent → 用户 | 卡片修改结果 |
| `error` | Agent → 用户 | 错误信息 |

### 5.3 响应格式

```json
{
  "status": "success" | "error",
  "data": { ... },
  "message": "操作成功/失败原因",
  "timestamp": "2026-06-01T12:00:00Z"
}
```

### 6.3 组件设计

#### 6.3.1 卡片展示组件 (CardView)

- 显示卡片标题、内容、案例、问题
- 内容中的关键词自动检测并渲染为超链接
- 底部显示来源信息和更新时间

#### 6.3.2 目录导航组件 (Sidebar)

- 按 `category` 分类展示卡片列表
- 可折叠/展开分类
- 点击卡片标题跳转到对应卡片
- 支持搜索过滤

#### 6.3.3 搜索栏组件 (SearchBar)

- 输入关键词实时搜索
- 显示搜索结果列表
- 点击结果跳转到卡片
- 搜索不到时显示"让 Agent 补全"按钮

#### 6.3.4 导航控制组件 (NavControls)

- **前进按钮：** 跳转到浏览历史中的下一张卡片
- **后退按钮：** 跳转到浏览历史中的上一张卡片
- 使用浏览历史栈实现

#### 6.3.5 对话窗口组件 (ChatPanel)

- 可折叠的底部对话面板

## 8. 关键设计决策

### 8.1 为什么选择 ChromaDB 而非关系型数据库？

- **语义搜索需求：** 知识问答的核心是语义相似度检索，向量数据库天然支持
- **本地轻量：** ChromaDB 无需独立服务，嵌入 Python 进程即可运行
- **持久化简单：** 自动持久化到本地磁盘，无需额外配置

### 8.2 为什么选择 LangGraph 而非纯 LangChain？

- **有状态工作流：** LangGraph 的 StateGraph 天然支持多步骤、有状态的 Agent 工作流
- **条件分支：** 支持"信息不足时搜索补充"等条件判断
- **可观测性：** 每个节点的输入输出可追踪，便于调试

### 8.3 为什么选择 DaisyUI 而非 React/Vue？

- **零构建工具链：** 纯 HTML + CDN 引入，无需 npm/webpack
- **本地运行：** 用户只需打开 HTML 文件即可使用
- **快速开发：** DaisyUI 提供丰富的预制组件

### 8.4 卡片关联检测策略

- **检测时机：** 卡片入库时和内容修改时
- **检测范围：** 所有已存在的卡片标题和别名
- **匹配方式：** 精确匹配 + 模糊匹配（包含关系）
- **关联更新：** 新增/修改卡片时，自动更新相关卡片的 `related_cards`

## 9. 安全与隐私

- **本地运行：** 所有数据存储在本地，不上传云端
- **API Key 管理：** OpenAI API Key 存储在本地环境变量或配置文件中
- **文件隔离：** 上传文件存储在 `uploads/` 目录，不对外暴露
